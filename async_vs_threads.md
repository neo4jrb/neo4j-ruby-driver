The public APIs are identical. jRuby wrapper is here to stay. Let's talk now only about the mri part. Is async gem the right decision or threads would be better ?

For a driver, the answer is threads — but let me reframe it slightly, because "Async vs threads" is the wrong frame for *library* code, and the right frame makes the decision clear.

A driver should not pick the host application's concurrency model. That's the whole principle. The people embedding your driver are overwhelmingly running threaded hosts — Puma under Rails, Sidekiq workers — none of which have a fiber reactor running. If you build the driver *on* Async, you're forcing a reactor onto users who didn't ask for one, and it composes badly: either you require the whole app to be Async (Falcon-and-down), or you spin up a reactor inside a thread and deal with the nested-model mess. That's a lot of imposition for a query client.

So concretely: build the driver on **plain blocking socket I/O plus a thread-safe connection pool**, and don't depend on Async at all. The host's threads provide the concurrency — each thread checks out a connection, runs its Bolt exchange, checks it back in. This is exactly the pg / mysql2 / redis-rb model, and it drops cleanly into every threaded host that's actually deploying Neo4j.

Here's the part that makes this strictly the better call rather than just the safe one: because your transport is pure Ruby, blocking I/O + a thread-safe pool is *also* Async-compatible for free, with zero Async code. On MRI, standard blocking I/O methods are scheduler-aware — under a fiber scheduler they auto-yield to the reactor. And `Mutex`/`ConditionVariable` became per-fiber and scheduler-aware in Ruby 3.0, so a normal thread-safe pool checkout (the `connection_pool` gem's `Mutex` + `ConditionVariable`) yields correctly under Async too. So the same neutral implementation runs concurrently on threads *and*, if someone deploys it under Falcon, runs non-blocking on fibers. You get the Async crowd without an Async dependency or a second adapter to maintain.

Building *on* Async would actually buy you nothing here that the pool doesn't already give. Async's headline win is thousands of cheap fibers — but a driver caps concurrency at pool size anyway, because you don't want thousands of live connections to Neo4j. Your in-flight query count is bounded by the pool, not by threads-vs-fibers. So the fiber advantage is mostly irrelevant to a pooled client.

A few sharp edges to keep the "free Async compatibility" property intact, since you'd lose it quietly if you trip them:

- All I/O must go through Ruby's own methods. You're pure Ruby, so you're fine — but don't introduce a C-extension that does its own socket reads (a native msgpack/Bolt-pack lib, say), or it'll bypass the scheduler hooks and block the reactor under Async. On JRuby it wouldn't matter, but this is the MRI path.
- DNS is the classic one that doesn't yield. `getaddrinfo` is blocking and historically not scheduler-aware; modern MRI added an `address_resolve` hook, but it's the place a "non-blocking" driver secretly stalls the reactor. Worth being aware of if you care about Falcon users.
- The pool's checkout has to be the thing that blocks when exhausted, via Mutex/ConditionVariable, not a busy-loop — which `connection_pool` already does correctly, and which is what makes it dual-mode.

And this is the same lever behind the Sidekiq pool-saturation pain: with a bounded pool, the real tuning knob is pool size vs. host concurrency (threads × processes), and that's true whether the host uses threads or fibers. Threads-vs-Async doesn't change it; pool sizing does.

So: don't build on Async. Build neutral — blocking I/O on a thread-safe pool — and you serve threaded hosts as the default and Async hosts for free, with one implementation.

The driver has a pipelining feature where requests writes and response reads should be asynchronous. E.g. the driver should pull records in the background and fill records to a buffer up to a watermark before the client explicitly requests them. What architecture would you use for that?

Great problem — this is where a driver actually earns its keep, and it's worth being deliberate about the layering because "do it in a background thread" is the tempting-but-wrong first instinct.

The architecture I'd use is **sans-I/O protocol core + a pluggable pump + a bounded record buffer with high/low watermarks.** Concretely, four layers:

1. **Sans-I/O Bolt core.** A pure state machine: feed it received bytes and it emits messages (RECORD, SUCCESS{has_more}, FAILURE); ask it for the next bytes to send and it hands you the serialized RUN/PULL/DISCARD. It owns the response-ordering FIFO (Bolt replies in request order, so each sent request pushes an expected-response handler). No socket anywhere in it. This is the piece that's fully unit-testable without a network and, not incidentally, the piece that stays identical regardless of how you drive I/O.

2. **Record buffer with watermarks.** Holds decoded records for one result stream, tracks consumed count, and owns the autopull *policy*: when buffered drops below the low watermark and `has_more` is true, enqueue the next `PULL {n: fetchSize}` into the core; when it's at the high watermark, withhold. This is where your watermark lives — keep it out of the I/O layer.

3. **The pump (the only concurrency-aware part).** A loop that moves bytes both directions: read socket → feed core → route emitted records into the buffer; drain core's outgoing bytes → write socket. Everything about "background" is isolated here.

4. **Cursor/result API** on top — `next`/`each`/`peek`/`close` — popping from the buffer, signaling demand, blocking on empty until the pump fills or the stream ends. Its lifecycle owns the pump and the checked-out connection.

The central decision you're really asking about is what drives the pump, and here's the recommendation: **make true background prefetch a pump variant, not a baked-in assumption.** Because the core is sans-I/O, "background" vs "on-demand" is a swap of the outer loop, not a rewrite:

- **On-demand pump** (default, zero extra threads): runs on the consumer's thread. When `next` empties the buffer, it reads the next batch off the socket into the buffer and returns one. The buffer is still "filled ahead" relative to individual `next` calls — you just don't pay for a background context. This stays fully neutral to the host's concurrency model, which is the property you wanted to preserve from the pooling decision.
- **Prefetch pump** (opt-in, true background): runs in its own execution context per streaming result, filling the buffer to the high watermark ahead of demand, so `next` hits memory with no socket read and even the batch-transfer latency is hidden. This is the one place the driver legitimately needs its own concurrency primitive — so abstract it behind a tiny `Executor`: a `Thread` by default, a fiber when a fiber scheduler is present. Same pump code; blocking I/O auto-yields under Async, runs parallel-ish under threads.

That keeps faith with the earlier conclusion: the driver doesn't impose a model on the host except in the explicitly opted-into prefetch case, and even there the executor is swappable.

On backpressure, be clear that there are two levels and they're different:

- **Server→driver** is Bolt flow control: `fetchSize` + `has_more` + watermark-gated PULLs. You never get unbounded data because you simply don't send the next `PULL` until the buffer drains below the low watermark. Use hysteresis here, not a per-record trigger — high watermark around 1–2× `fetchSize`, low watermark a fraction of it — or you'll thrash PULLs, sending a new batch request every time one slot frees. The batch is your natural granularity since you can't stop the server mid-batch anyway.
- **Driver→consumer** is the in-process handoff. A `SizedQueue` is the right primitive: the producer (pump) blocks when it's full, the consumer pops, and the bound *is* the backpressure. The watermark/PULL-pacing logic sits just above it, deciding when to ask the server for more.

Two hazards worth designing for up front. First, the socket: let the pump own the read side and issue the PULLs itself, so you have a single reader and serialize writes behind one mutex (the consumer thread may also write — opening a new query, sending DISCARD). One reader, guarded writer; don't let two threads read the same socket. Second, lifecycle: the prefetch pump is tied to the cursor, not the connection. On early `close`, send DISCARD and drain to the stream boundary before the connection returns to the pool, or you'll hand a dirty connection back. Exhaustion does the same cleanup naturally.

If reader-thread count ever becomes a concern — many concurrent streaming results, each with its own prefetch thread — the escalation is a single shared I/O thread running a selector (`IO.select`/`read_nonblock`) that demultiplexes all connections into their buffers. It scales to one thread regardless of pool size, but you're now building a small reactor, so I'd only reach for it if profiling says the per-stream threads hurt. For pool-bounded concurrency (at most `pool_size` streaming connections), per-stream pump threads are modest and much simpler.

Net: sans-I/O core so the protocol logic is concurrency-agnostic and testable, watermark/autopull policy in the buffer layer, and a pump you can run on-demand (neutral default) or as a background Thread/fiber (opt-in prefetch). That gives you the asynchronous write/read decoupling and watermarked background fill without committing the whole driver to a reactor.

I would like to come back to the sans-I/O protocol core + a pluggable pump + a bounded record buffer with high/low watermarks architecture. With the prefetch pump would you have an executor thread per client thread, per connection or have a single thread which multiplexes between connections?

Per connection — and specifically per *active streaming result*, since a Bolt connection carries exactly one ordered stream at a time. That's the default. Per-client-thread is the wrong axis, and the single multiplexing thread is a scale escalation you reach for only under profiling pressure, not by default. Let me walk the reasoning, because the *why* rules two of the three out cleanly.

Per-client-thread is wrong because the pump should bind to the resource it pumps — the socket and its ordered response stream — not to the consumer. Think about what a client thread actually is to the driver: it holds a connection only while it's streaming, and nothing in between. While it's actively iterating a cursor, it maps 1:1 to exactly one connection anyway, so per-client-thread buys you nothing over per-connection during streaming. What it adds is cost: idle pump threads for client threads that aren't streaming, a re-targeting lifecycle as a long-lived pump has to follow its client across different checkouts and different queries, and — worst — the risk of the client thread and its pump thread both touching the same socket. The unit of work here is unambiguously the connection's stream, so bind there.

Between per-connection and the single selector, default to per-connection for simplicity, and note that it's not just the natural granularity but the *maximal correct* one: Bolt responses are strictly ordered on a connection, so a single reader must process them in sequence. Two readers on one socket would corrupt ordering. You can't go finer than one-per-connection even if you wanted to. And one-per-connection gives you a set of completely independent producer-consumer pairs — each pump owns one socket's read side, one bounded buffer, one watermark policy, blocking I/O, local backpressure via a `SizedQueue`. No shared mutable state between pumps, no cross-connection coordination, no interest-set bookkeeping. The pump stays the trivial, swappable component the sans-I/O design was meant to keep it.

The thread count objection is the obvious pushback, and here's the sharp point that defuses it: the count is bounded by pool size, because you can't have more active streaming connections than the pool hands out. For a normal pool (10–50) that's a modest number of background threads. And critically — long-lived blocking pumps don't benefit from a shared bounded executor, which is the middle option people reach for to cap threads. A prefetch pump occupies its thread for the whole duration of the stream (it's blocked on I/O most of the time). So if you size an executor smaller than the pool, some streaming connections simply can't prefetch concurrently; to let all of them prefetch at once you need ≥ pool_size pump threads regardless. "Thread per active stream" and "executor sized to pool_size" are the same thing. The executor framing doesn't reduce the count — it just adds a layer. So the real choice collapses to binary: thread-per-stream (simple, ~pool_size threads) or a single cooperative loop (one thread, reactor complexity). There's no free middle.

The single multiplexing thread is that other pole, and it's a genuine reactor: one I/O thread running `IO.select` over all active sockets, nonblocking reads, demuxing bytes into each result's sans-I/O core, and — the part people underestimate — managing read interest per connection for backpressure. When one result's buffer hits the high watermark you have to *deregister that socket from the select set* while continuing to service the others, then re-register when it drains to the low watermark. That's exactly the interest-set management of an event loop, plus per-connection partial-read state, per-connection error/timeout handling, and dynamic add/remove as connections check in and out. The sans-I/O core makes it *feasible* — feed bytes, get events is precisely its job — but you are now building and maintaining a slice of the cooperative scheduler we deliberately decided the driver shouldn't depend on, except now it's internal and mandatory. Only take that on if profiling says the per-stream thread count or the context-switch overhead is a real cost — which means large pools with many simultaneous live streams. And at that scale, the honest question is whether the host should just be running Falcon/Async and you should ride *its* reactor through the colorless-I/O path, rather than hand-rolling your own.

One thing that's the same under both models, so it shouldn't drive the decision: under MRI's GVL, the only thing prefetch overlaps is the socket wait against the client's processing of already-buffered records, and both models achieve that overlap. Record decoding is GVL-serialized either way — a single selector funnels all decode through one thread, and per-connection threads are still serialized by the GVL for the Ruby parts. So this is a thread-overhead-versus-reactor-complexity decision, not a parallelism one. Don't pick the selector expecting throughput from parallel decode; you won't get it on MRI.

So: per-connection pump, owning both the reads and the PULL-writes for its socket (single owner, client thread touches only the buffer), lifetime scoped inside the connection's checkout and torn down before it returns to the pool. Keep the selector in your back pocket as the one escalation, gated on a measured thread-count problem, and know that taking it means owning a real event loop.

So the thread should not be parked along the connection in the pool but freshly creates at connection acquisition and torn down at connection release?

Not quite — I'd push back on tearing it down at release. Bind the pump thread's lifetime to the *connection's pooled lifetime*, not to the individual checkout. Create it lazily, park it between streams, and only destroy it when the connection itself is evicted or the pool closes. Per-checkout create-and-destroy is the version I'd avoid as the default.

The reason is hot-path churn. A driver executes a lot of queries, and `Thread.new` + `join` on every checkout means a `pthread_create` and a teardown per query. That's real latency on each call plus allocator and OS-scheduler pressure, and it's entirely avoidable. On top of that, most checkouts may not even stream — a small result or a write needs no prefetch at all — so you'd be spinning up and joining threads you barely use. Creating a thread is the kind of cost driver authors specifically engineer out of the request path.

The part of your instinct that's correct is that the pump must do *nothing* while the connection sits idle in the pool — you don't want a live reader running per parked connection. Agreed. But the way to get that isn't to destroy the thread, it's to *park* it: block it on a control queue or condition variable so it consumes no CPU and isn't touching the socket. Parked is not active. It costs you a little stack memory and nothing else, and it's bounded by pool size anyway, so for a normal pool (tens of connections) it's negligible.

So the lifecycle I'd run: on the first prefetch stream for a given connection, lazily spawn its pump thread; when a stream ends, the pump transitions active→parked and the connection returns to the pool with a sleeping reader attached; on the next checkout that streams, you hand the parked pump a fresh "drain this result into this buffer" job rather than building a new thread; and only when the connection is evicted (idle timeout, max-lifetime, pool shutdown) do you signal the pump to exit and join it. The thread's life tracks the connection's, begun lazily. Lazy creation also means connections that only ever do small reads or writes never grow a pump thread, which trims the resident count.

The one hard invariant, and it holds whichever way you go: the pump must be fully quiesced — parked, at the stream boundary, current batch drained or DISCARDed — *before* the connection goes back in the pool. Otherwise the next checkout can get a connection with a rogue reader mid-socket. With the park model that's just the active→parked transition gating the checkin; with your create/destroy model it'd be the join gating it. Either way there's a synchronization barrier at release.

If holding a parked thread per pooled connection bothers you at larger pool sizes, the leaner variant is a shared pump-thread pool sized to your *expected concurrent-streaming count* rather than to the full connection pool — since the number of connections streaming at any instant is usually far below pool size (it's bounded by how many client threads are mid-iteration). A streaming result submits a "pump me" job and adopts a free pump thread for the duration. You lose the clean socket affinity of the per-connection thread and you have to size the pool right, but you hold fewer resident threads when streaming is sparse. Both are reuse models; park-with-connection is simpler, the shared pool is more memory-frugal under low streaming concurrency.

All that said — if this driver's throughput is modest and you'd rather have the dead-simple mental model, per-checkout create-and-destroy is *acceptable*; it's correct, it just leaves performance on the table. It's a throughput-versus-simplicity call, and the reuse only pays off on the hot path. For a general-purpose driver I'd assume the hot path matters and reuse.

Ok, so a thread per connection for its lifetime. But let's play the scenario where the client runs within reactor. How does that look in such case?

Under a reactor, the per-connection pump stops being an OS thread and becomes a per-connection **fiber on the host's reactor**. That's the whole point of having made the pump a pluggable executor with a sans-I/O core: the loop body is byte-identical — feed socket bytes to the core, push decoded records to the bounded buffer, issue the next PULL when you cross the low watermark — and the only thing that swaps underneath it is the execution context. Thread in the blocking world, fiber in the reactor world.

The reason it composes is the colorless-I/O property we kept coming back to. The pump's "blocking" socket reads, when that pump runs as a fiber under a scheduler, hook into the scheduler's `io_wait` and *yield to the reactor* instead of blocking it — so while the pump fiber is waiting on the socket, every other fiber in the app keeps running. And the buffer handoff has to use scheduler-aware primitives (`Thread::SizedQueue`, `Mutex`, `ConditionVariable` — all integrated with the scheduler on MRI 3.x), so when the consumer fiber finds the buffer empty it *yields* rather than parking the reactor thread, and it gets woken when the pump pushes. Everything stays cooperative and on the reactor. Crucially, use the *standard* scheduler-aware primitives, not `Async::Queue` and friends — you want to bind to the Fiber scheduler interface, not to Async specifically, so the driver runs under any conformant reactor.

Now the part that actually changes your lifecycle decision: the "thread per connection for its lifetime, parked and reused" model was a *thread-cost* artifact. The only reason we parked and reused was that `pthread_create` on the hot path is expensive. Fibers don't have that cost — a fiber is a Ruby object and a stack, cheap to spin up and discard. So under a reactor the parking/reuse apparatus just evaporates: you spawn a fresh pump fiber per stream (per cursor) and let it die when the result is exhausted. No connection-lifetime binding, no parked-fiber pool, no wake-with-a-job machinery. The asymmetry is the takeaway — reuse-for-the-connection's-lifetime is the *threaded* answer; fresh-per-stream is the *reactor* answer, and both fall out of the same executor seam.

It's worth going one step further and questioning whether you want a separate pump fiber at all under a reactor. In the blocking world the dedicated reader earns its keep because on-demand reads would hard-block the client thread; the prefetch thread is how you overlap. But under a reactor, an on-demand read in the *consumer* fiber already yields cooperatively — interleaving is free. So you get most of the benefit just by keeping a PULL in flight (issue the next batch request at the low watermark, read the response lazily in the consumer fiber); the server works ahead, and the lazy read yields without stalling anything. A dedicated pump fiber adds only the read-*ahead-into-buffer* on top of that — hiding the decode too — which is a real but much smaller win than it was against blocking threads. So a perfectly reasonable reactor-mode design is *no separate pump*: in-fiber pipelining, and the bounded buffer plus watermarks still pace the PULLs. You'd reach for the dedicated pump fiber only if profiling shows the consumer's own decode latency is worth hiding.

Mechanically, the driver picks the mode by checking `Fiber.scheduler` at stream start: non-nil means schedule the pump as a fiber on the current reactor (or drop to in-fiber pipelining); nil means the OS-thread-per-connection path. That check is the *only* place the driver is concurrency-mode-aware — the sans-I/O core and the colorless reads make the two bodies the same.

Two caveats to keep the reactor path honest. First, the whole thing hinges on the transport going through Ruby's scheduler-hooked I/O. If any socket read happens inside a C extension that does its own syscall, it bypasses `io_wait`, hard-blocks the reactor thread, and freezes every fiber in the process — so pure-Ruby Bolt I/O isn't just a preference here, it's the precondition for reactor support. Second, the thing *not* to do is keep the pump as an OS thread while the consumer runs on a reactor. It technically works — the scheduler's `unblock` hook is callable cross-thread, so the pump thread pushing a record can wake the blocked consumer fiber — but you've now reintroduced one preemptive OS thread per connection into an app that chose a reactor precisely to avoid that, plus cross-thread synchronization on every record and GVL contention between the pump thread and the reactor thread. It runs; it just quietly defeats the reason the user is on a reactor. Under a reactor, the pump goes where the consumer is: on the reactor, as a fiber.

So can the fiber pump be implemented without requiring the async gem?

Yes — and this is the clean payoff of the whole "bind to the interface, not the gem" stance. The thing you depend on is the Ruby *core* fiber-scheduler interface, which has been in stdlib since 3.0. The `async` gem is just one *implementation* of that interface. The driver programs against the interface and never references Async's API.

Concretely, the three things the fiber pump needs are all core Ruby. You spawn it with `Fiber.schedule { pump_loop }` — that's the stdlib primitive for starting a non-blocking fiber on the current thread's scheduler, no gem involved. The socket reads inside the loop are plain Ruby I/O, which hook into the scheduler's `io_wait` and yield (the colorless property). And the buffer handoff uses `Thread::Queue`/`SizedQueue`, `Mutex`, `ConditionVariable` — all scheduler-integrated in core 3.x, so a fiber blocking on them yields to the reactor rather than parking the thread. You never need `Async::Queue` or `Async::Task`; the stdlib equivalents behave correctly under any conformant scheduler.

The framing that makes this work is that the driver is a scheduler *consumer*, not a *provider*. It rides whatever scheduler the host installed and never installs one itself. Async happens to be the implementation most people are running (because they're on Falcon or wrapped their app in `Async{}`), but the driver can't tell and shouldn't care — it works identically under Async, under any other conformant scheduler someone writes, and falls back to the thread pump when none is set. Installing its own scheduler is exactly what it must not do: that would either pull in async (defeating the question) or mean hand-rolling a reactor, and either way it'd be imposing a concurrency model on the host, which is the thing we ruled out at the very start.

Two honest caveats. First, "no async dependency" is not "fibers without a scheduler" — the fiber path only *does* anything when the host has set a scheduler, and `Fiber.schedule` actually raises if none is present. So you guard it behind the same `Fiber.current_scheduler` detection from before: scheduler present → `Fiber.schedule`; absent → thread pump. The driver supplies the fiber, the host supplies the reactor. Second, going stdlib-only means you give up Async's structured-concurrency conveniences and hand-roll the pump fiber's lifecycle yourself: cancellation on cursor close is cooperative (push a stop sentinel to its control queue and have it check between batches and unwind — there's no external "kill" for a scheduled fiber), and error propagation is manual (catch in the pump, stash the exception on the cursor/buffer, re-raise it in the consumer on its next `next`). That's a bit more bookkeeping than `task.stop` and task-tree error propagation would give you, but it's straightforward, and it's the price of staying gem-free and scheduler-agnostic.

Precondition is just the usual two: a recent-enough MRI (3.x) so the stdlib primitives are actually scheduler-aware, and pure-Ruby socket I/O so the reads go through the hooks rather than a C extension that would block the reactor. Given those, the statement is clean — the fiber pump depends on Ruby, not on Async. The gem, if it's there at all, belongs to the application.
